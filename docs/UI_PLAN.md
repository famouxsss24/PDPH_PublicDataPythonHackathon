# 그늘로 프론트엔드 전면 개편 계획 (v2 확정)

> **확정일: 2026-07-15.** 이 문서가 프론트·백엔드 공통의 단일 기준이다.
> 백엔드는 §7 API 계약을 구현한다. 구 UI(β슬라이더·양쪽 패널·시간 버튼)는 전면 폐기.
> 도메인 원칙: **시민용 그늘 경로 내비가 유일한 구현 대상.** 행정용(그늘막 입지·SDI)은 제안서에 글로만.

---

## 0. 설계 원칙

1. **지도가 주인공, UI는 조연** — 흰 불투명 카드 + 여백 + 약한 그림자. 선·테두리·글래스 제거 (전역 규칙 13)
2. **간지는 그림자 표현 하나에만** — 나머지는 네이버지도/카카오맵 문법 그대로. 학습 비용 0
3. **프론트 먼저, 백엔드는 계약을 따라옴** — 전 엔드포인트 stub(mock JSON)으로 UI 100% 완성 후 백엔드가 §7 구현
4. **화면의 모든 숫자·선·시각이 단 하나의 상태를 가리킴** — Codex P0-1/P0-2 근본 해결
5. **모든 것은 목적지에서 시작** — 목적지를 찍으면 출발지는 내 위치로 자동 설정

- **A→B 전환**: 목적지 선택 즉시 출발지=내 위치(GPS) 자동 설정 → `depart_at=지금` 기준 경로 옵션 자동 계산
  - GPS 거부/실패/노원 영역 밖 → "출발지를 지도에서 선택해주세요" 폴백. 출발지 변경·⇄교환은 B의 명시적 버튼
- **경로 옵션 5종, 순서 고정**: ①추천(그늘+시간 균형) ②최단 ③그늘 우선(최대) ④그늘 30%+ ⑤그늘 50%+
  - 카드 정보 위계: 1순위 **소요시간(분)+도착시각**, 2순위 그늘 %, 3순위 +추가시간. `<button>` 라디오그룹 (키보드 선택 가능)
- **Bottom sheet 3단 스냅** (peek 96px / half 45dvh / full 88dvh), drag handle, `100dvh`+`safe-area-inset`
  - 시트 높이 변경 시 `ResizeObserver`로 지도 fitBounds padding 재계산 (Codex P0-3 해결)
- 경로 생성 후 지도 오클릭은 무시 (전체 초기화 사고 제거)
- 데스크톱 = 좌측 380px 단일 패널(검색↔결과 단계 전환) + 지도 전부. 양쪽 패널 구조 폐기

## 2. 시간 = 입력 패러다임

**"최적 출발시각 추천"이라는 별도 기능은 없다. 시간은 경로 탐색의 입력값이다.**

- 기본값 `depart_at = 지금`. B 상단 `[지금 출발 ▾]` 칩 → 네이버식 시간 선택기
- **시간 선택기 배경에 햇빛 노출 곡선** (10분 단위) + 최적 시각 마커 — 구 departure 기능의 환생

```
┌──────────────────────────────┐
│  출발 시간               ✕    │
│  ▁▂▃▅▇▇▅▃▂▁  ← 시간대별 노출  │
│  07    12    19               │
│        ▲최적 16:40            │
│  [지금] [14:00] [14:10] …     │
└──────────────────────────────┘
```

- **추천 출발 미니 칩**: 시간 칩 옆 `추천 출발 16:40` 텍스트 — 작고 조용하게(배경·테두리 없음). 누르면 그 시각으로 재계산 (시간 선택기에서 고른 것과 동일 코드 경로)
- 시각 변경 → 태양각·건물 그림자 재계산 → 같은 5옵션 재제시
- 일몰 이후(19시~) 선택: "밤에는 그늘 구분이 없어요" 안내 + 최단만
- 계산 기준일 = CALC_DATE(8/6) 고정 — meta에 정직하게 표기
- 규칙 14: 시간 칩·미니 칩은 최장 문자열("내일 14:30 출발") 기준 폭 예약, tabular-nums

## 4. 간지 포인트 — 그림자에만 투자

1. **경로선 세그먼트 색칠** ★핵심 — 경로선을 구간별 주황(햇빛)/남색(그늘)으로. "왜 그늘 경로인가"를 선 하나가 설명. 옵션 카드의 미니 그늘 비율 바와 색 일치
2. **시간 스크럽 그림자 스윕** (D 상태) — 프레임 사이를 크로스페이드하고 시계에 현재 전환 시각을 표시 (Codex 10.3)
3. **3D 그림자뷰** — 진입 시 메인에서 선택한 시간·경로(option_id)로 정지 시작 → 사용자가 재생. 태양 고도 나침반 동반

## 5. 성능 아키텍처

### 5-1. 전 엔드포인트 stub
- `web/mock/*.json` (places·routes·exposure·shade 시간별·steps) + `api.js`의 `USE_MOCK` 플래그
- 백엔드 없이 전체 UI 개발·데모 가능. 발표장 네트워크 장애 시 mock 폴백으로 방어

### 5-2. 스케일(줌) 구분 렌더 — LOD
| 줌 | 그림자 | 격자 |
|---|---|---|
| <14 | 강한 simplify 사전생성본 | off |
| 14–16 | 표준 | canvas, 툴팁 이벤트 위임 |
| 16+ | 풀 디테일 | 상동 |

LOD별 산출물은 파이프라인 사전생성 (런타임 simplify 금지). API `lod` 파라미터.

### 5-3. 뷰포트 lazy loading + 경합 방지
- `moveend` debounce 250ms → 현재 bbox만 요청, 라운딩 bbox 키로 클라이언트 캐시
- 모든 fetch에 `AbortController` + request token — 최신 요청만 화면 반영
- 타임랩스: 재생 전 프레임 prefetch → 준비 후 재생

## 6. 상태 관리 + 파일 구조

```
state.js — 단일 상태 객체, action 함수만이 상태 변경:
  setDest → (자동 setOrigin=내위치) → requestRoutes
  selectRoute(optionId) · setDepartAt(t) · enterShadowExplore/exit · reset(scope)
```

- `selectRoute(optionId)` **하나**가 카드·지도 선·요약·턴바이턴·3D 링크 전부 갱신 (P0-1 소멸)
- 그림자 탐험 시간 ≠ 경로 계산 시간 상태 분리 (P0-2 소멸)
- `reset`: "경로만 지우기" vs "전체 초기화"(watchPosition 해제 포함) 구분

```
web/
├─ index.html  ·  3d.html
├─ css/  tokens.css · layout.css · components.css
├─ js/   api.js · state.js · map.js · sheet.js · search.js
│        routes.js · steps.js · shade.js · timelapse.js · voice.js
└─ mock/ *.json
```

buildless ES modules (`type=module`, CDN 유지, 빌드 도구 없음). SPEC "1파일" 규칙 해제 — SPEC·PROJECT_STATUS 동기화 필요.

## 7. API 계약 v2 (백엔드 구현 명세 — 프론트가 정의)

```
GET /api/routes?from&to&depart_at=now|HH:MM
 → { options: [ 추천, 최단, 그늘최대, 그늘30, 그늘50 ] 순서 고정,
     각 옵션: { id, minutes, arrive_at, distance_m, shade_pct, extra_min,
               segments: [{coords, shaded:bool}],   ← 세그먼트 색칠용
               steps: [{text, dist_m, turn}] },
     recommended_id, snapped: {from, to},
     meta: { depart_at_used, sun: {alt, az}, calc_date, note } }

GET /api/exposure?from&to        → 시간 선택기 곡선 (10분 단위 + best_time)
GET /api/shade?hour&bbox&lod     GET /api/grid?bbox&lod
GET /api/places?q                GET /api/health
GET /api/brief?option_id         → AI 브리핑 (§8-A)
POST /api/parse_query {text}     → 자연어 검색 파싱 (§8-C)
```

**백엔드 필수 반영:**
- 임의 depart_at = 사전계산 5시간대 edge 그늘값의 10분 단위 보간 (기존 departure 로직 재사용)
- 경로를 `(u,v,key)` edge 목록으로 보존 — 수치·스텝·세그먼트가 같은 edge에서 산출 (Codex P0-4)
- 좌표 범위·노원 경계 검증 422 + 스냅 좌표 반환
- 데이터 없어도 서버 기동, `/api/health`로 상태 노출, 미준비 기능만 503
- 전 응답 GZip + ETag/Cache-Control (GZip은 구현 완료)

## 8. AI 차별화 (확정: A+C / B는 스파이크 조건부)

### A. AI 그늘 브리핑 + 음성 보행 코치 ✅ 확정
- 경로별 LLM(Haiku) 1회 호출 — 세그먼트 데이터 근거 한 줄 브리핑: "처음 300m는 뙤약볕, 가로수길부터 쭉 그늘"
- 길안내 중 Web Speech API(TTS) 음성 코칭. 발표장 대비 mock 브리핑 폴백
- 숫자 못 읽는 고령층에게 음성으로 — 취약계층 서사와 접합

### C. 자연어 검색 ✅ 확정 (A의 부속)
- 검색창이 자연어 수용: "4시에 애들이랑 도서관 갈 건데 안 덥게" → {목적지, depart_at=16:00, 그늘 선호} 파싱 → 시간 선택까지 자동
- function calling 1회, 실패 시 일반 검색 폴백

### B. 거리뷰 × AI 비전 그늘 교차검증 ⚠️ 스파이크 조건부
- 착수 전 1시간 스파이크: 카카오 로드뷰 이미지를 코드로 확보 가능한지만 검증 → **불가면 즉시 드랍**
- 가능 시: 대표 경로 2~3개 오프라인 배치 — 비전 LLM이 아케이드·캐노피·수목 판독, 지도 "AI 검증" 배지 + 제안서 그래픽. 라이브 기능 아님

**공통 원칙: AI는 프론트 개편 1~3단계 완료 후에만 착수.** 미완성 내비 위 AI는 장식이다.

## 9. 구현 순서 (각 단계가 그 자체로 데모 가능)

1. **뼈대** — 파일 구조·토큰·지도·검색·bottom sheet 상태머신 (mock)

4. **안내** — 턴바이턴 + 음성 코치(A)
5. **그림자 간지** — LOD·prefetch·타임랩스·3D 정합
6. **AI** — 브리핑(A)·자연어 검색(C)·B 스파이크
7. **백엔드 통합** — USE_MOCK off, §7 계약 검증
8. **폴리시** — 360×800 포함 뷰포트 6종 스크린샷 검증(playwright), reduced-motion, 오류 상태 문구
