# 그늘로 구현 가이드와 학습 노트

> 처음 저장소를 보는 팀원과 발표자가 현재 제품을 정확히 설명하기 위한 문서
> 최종 갱신: 2026-07-16

## 1. 서비스 한 줄 설명

그늘로는 노원구 안에서 출발지와 목적지를 검색하고, 건물 높이·가로수·태양 위치로 계산한 그늘을 이용해 햇빛 노출이 적은 도보 경로를 비교하는 웹 내비게이션이다.

## 2. 사용자가 보는 흐름

1. 상호명, 건물명, 행정구역명, 지번·도로명 주소로 출발지와 목적지를 검색한다.
2. 장소를 고르면 지도가 해당 위치로 이동해 선택이 맞는지 보여준다.
3. 두 지점이 정해지면 경로가 순서대로 그려지고 최대 5개 경로 비교 화면이 열린다.
4. 시간·거리·그늘 비율을 보고 경로를 고르면 도보 안내로 전환된다.
5. 실제 이동 때는 GPS를 따라가고, `경로 미리보기`를 누른 경우에만 경로를 자동 재생한다.
6. 3D 화면에서는 높이별 건물과 시간에 따라 바뀌는 그림자를 근거 화면으로 확인한다.

검색이 기본이고 지도 클릭은 보조 수단이다. 출발·도착 바꾸기, 다시 입력, 라이트·다크 모드, 출발 시각 설정을 지원한다.

## 3. 화면에서 구현된 것

### 검색과 경로 비교

- Kakao Local 검색을 우선하고 OSM·공공 주소 로컬 색인으로 폴백한다.
- 선택한 장소는 지도 카메라 이동으로 확인한다.
- 최단, 추천, 그늘 우선 등 의미가 겹치지 않는 경로를 비교한다.
- 경로선의 그늘 구간과 햇빛 구간을 색으로 나누고 계산 중·화면 전환 모션을 제공한다.

### 도보 안내

- 작은 반투명 부채꼴은 사용자가 바라보는 방향, 중앙 점은 현재 위치를 뜻한다.
- 다음 좌·우회전 지점은 지도 위 원형 회전 아이콘으로 표시한다.
- 다음 회전까지 거리, 남은 거리·시간, 주변 건물·상호 라벨을 갱신한다.
- GPS 추적과 경로 미리보기를 분리했고, 미리보기에서 실시간 안내로 즉시 돌아갈 수 있다.
- 모바일에서 지도를 직접 움직이면 추적을 잠시 멈추고 재중심 버튼 또는 일정 시간 뒤 복귀한다.
- 음성 버튼을 켜면 거리 구간별로 한 번씩 한국어 안내를 읽고, 같은 문장을 `role=status` 접근성 영역에도 전달한다.

### 3D와 그림자

- 건물 footprint를 실제 높이 또는 층수 기반 높이로 압출한다.
- 07~18시 그림자 프레임을 교차 페이드하고 태양 조명 방향도 함께 바꾼다.
- 첫 화면에는 현재 시각 프레임을 먼저 띄우고 나머지는 유휴 시간에 불러온다.
- 현재 표현은 높이 기반 도시 모델이지 사진 재질이 있는 게임 엔진급 디지털 트윈은 아니다.

## 4. 계산은 어떻게 이루어지는가

### 그림자

2026-08-06 맑은 하늘을 가정한 태양 고도·방위각과 건물 높이로 그림자 길이와 방향을 계산한다. 건물 footprint와 이동된 footprint의 convex hull을 사용하고, 가로수는 반경 3m 원으로 근사한다. 따라서 실제 날씨와 실시간 관측 그림자가 아니다.

### 경로

OSM 보행 그래프의 각 엣지에 거리와 시간대별 그늘율을 붙인다. 출발 시각에 맞춰 그늘율을 보간한 뒤 거리와 햇빛 노출 비용의 가중치를 바꿔 여러 경로를 계산한다. 이 과정은 수학적 최적화이며 생성형 AI가 아니다.

### 내비게이션

경로 좌표를 누적 거리 축으로 바꾸고, 턴바이턴 단계의 거리 합을 실제 경로 길이에 맞춰 정규화한다. 현재 진행 거리와 다음 회전 거리를 같은 축에서 비교해 회전 마커와 음성 안내 시점을 정한다.

## 5. 현재 AI 활용 상태

현재 제품 경로에는 실제 LLM·비전 모델·학습 모델 호출이 없다.

| 항목 | 현재 상태 | 정확한 설명 |
|---|---|---|
| 그늘 경로 추천 | 구현 | 물리 시뮬레이션과 가중 최단경로 계산. AI 모델 아님 |
| 경로 인사이트 문장 | 구현 | 경로 수치에 따른 규칙 기반 템플릿. LLM 생성 아님 |
| 한국어 음성 안내 | 구현 | 브라우저 Web Speech API가 기기 음성을 재생. 앱의 AI 모델 아님 |
| AI 그늘 브리핑 `/api/brief` | 계획 | 아직 API와 모델 호출 미구현 |
| 자연어 조건 파싱 `/api/parse_query` | 계획 | 아직 미구현 |
| 거리뷰 비전 검증 | 조건부 계획 | 라이선스·평가셋·정확도 검증 전에는 제품 기능으로 주장하지 않음 |
| LightGBM·SHAP | 선택 의존성/계획 | 현재 실행 경로와 발표 결과에 포함되지 않음 |

경진대회에서 지금 상태를 “AI 경로 추천”이라고 부르면 과장이다. AI 항목을 사용하려면 실제 모델 입력·출력, 실패 시 폴백, 평가 지표와 재현 가능한 결과가 먼저 추가되어야 한다.

## 6. 음성 안내 설계

- 안내 문장은 `210미터 앞에서 우회전하세요`, `잠시 후 좌회전하세요` 형식으로 조합한다.
- `골목길`처럼 정보가 없는 도로명은 음성에서 생략한다.
- 기기에 여러 한국어 음성이 있으면 Natural, Online, SunHi, Heami 등 자연음성 후보를 우선한다.
- 현재 개발 PC에는 `Microsoft Heami` 한 종류만 있어 음색 품질은 운영체제 음성에 제한된다.
- 상용 지도 앱의 녹음 음성을 추출해 포함하지 않는다. 저작권과 배포 허가를 확인할 수 없기 때문이다.
- 모든 기기에서 같은 고품질 음성이 필요하면 정식 라이선스가 있는 클라우드 신경망 TTS를 백엔드에 연결하고 캐시·비용·개인정보 정책을 별도로 설계해야 한다.

## 7. 코드 구조

백엔드 전체가 미구현인 상태는 아니다. FastAPI의 `/api/health`, 장소 검색·역지오코딩·주변 장소, 복수 경로, 출발 시각 추천, 건물·그림자·태양 위치 API는 구현되어 있다. 현재 미구현인 백엔드는 AI 전용 `/api/brief`, `/api/parse_query`와 실제 모델 실행 부분이다.

| 모듈 | 책임 |
|---|---|
| `state.js` | 화면 전체의 단일 상태와 action |
| `search.js` | 검색, 자동완성, 출발·도착 편집 |
| `routes.js` | 경로 카드, 선택, 규칙 기반 인사이트 |
| `navigation.js` | GPS·미리보기, 진행률, 다음 회전 판정 |
| `steps.js` | 턴바이턴 화면, 음성, 접근성 알림 |
| `map.js` | 지도 레이어, 경로·현재 위치·회전·장소 마커, 추적 카메라 |
| `shade.js` | 2D 그림자 프레임 재생과 lazy load |
| `3d.js` | 3D 건물, 태양 조명, 그림자와 경로 러너 |
| `api.js` | 요청 취소·캐시와 live/mock 응답 정규화 |

내비게이션에서 위치 계산, 안내 문장, 지도 표현을 서로 다른 모듈로 분리해 한 파일이 GPS·DOM·음성을 동시에 관리하지 않게 했다.

## 8. 데이터와 한계

- 서비스 범위는 노원구 보행망이다.
- `web/mock`은 손으로 만든 가짜 데이터가 아니라 실제 파이프라인 산출물에서 추출한 발표용 정적 스냅샷이다.
- Kakao Local이 비활성화되거나 네트워크가 끊기면 최신 소규모 상호 검색률이 낮아진다.
- GPS가 경로에서 300m 이상 떨어져 있으면 임의로 경로에 붙이지 않고 경로 근처에서 시작하라고 표시한다.
- 저사양 실제 Android·iOS, 스크린리더, 색각 이상 사용자를 대상으로 한 실기기 검증은 아직 남아 있다.

## 9. 실행과 검증

```powershell
.venv\Scripts\python.exe -m uvicorn api.main:app --host 127.0.0.1 --port 8000
.venv\Scripts\python.exe -m ruff check .
.venv\Scripts\python.exe -m pytest -q
Get-ChildItem web\js\*.js | ForEach-Object { node --check $_.FullName }
```

브라우저 주소는 `http://127.0.0.1:8000`이다. 기능 변경 후 데스크톱과 390×844 모바일 화면에서 콘솔 오류와 UI 겹침을 확인한다.

## 10. 학습 노트

### 좌표와 회전은 같은 기준축을 써야 한다

GPS 거리, 경로 geometry 길이, API 단계 거리의 합은 조금씩 다를 수 있다. 서로 다른 값을 직접 비교하면 회전 아이콘이 교차로와 어긋난다. 단계 거리 합을 geometry 누적 거리에 맞춰 한 번 정규화하면 현재 위치, 다음 회전, 남은 거리를 같은 축에서 계산할 수 있다.

### 현재 방향과 다음 행동은 다른 정보다

부채꼴은 사용자가 바라보는 방향이고, 회전 아이콘은 앞으로 해야 할 행동이다. 두 정보를 하나의 큰 화살표로 합치면 사용자가 지금 방향과 다음 회전을 혼동한다. 역할과 크기를 분리해야 한다.

### 음성 품질은 엔진과 문장을 함께 봐야 한다

좋은 음성 엔진도 `골목길에서 우회전` 같은 조각 문장을 읽으면 어색하다. 정보가 없는 도로명을 제거하고 거리 표현에 `앞에서`, 동작에 `하세요`를 붙이는 것이 먼저다. 그 다음 기기에서 제공하는 자연음성을 우선 선택한다.

### 접근성 알림과 실제 소리는 별도 채널이다

Web Speech 음성은 사용자가 켜야 하지만, 다음 회전 변경은 스크린리더가 읽을 수 있어야 한다. 화면에 보이지 않는 상태 영역을 두면 소리를 끈 사용자도 보조기술로 같은 정보를 받을 수 있다.

### 3D 최적화는 첫 장면의 우선순위 문제다

모든 시간대 그림자를 먼저 받으면 3D 진입이 늦어진다. 건물·경로·현재 프레임만 병렬로 먼저 받고 나머지는 유휴 시간에 준비하면 같은 데이터량이어도 체감 로딩이 짧다.

### 물리 계산과 AI를 구분해야 한다

복잡한 계산이라고 모두 AI는 아니다. 현재 강점은 설명 가능한 물리 계산과 경로 최적화다. AI를 추가할 때도 기능 이름보다 검증 가능한 입력·출력과 실패 처리부터 설계해야 한다.
