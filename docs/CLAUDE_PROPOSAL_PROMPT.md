# Claude 제안서 작성 요청

## 역할

너는 공공데이터·공간정보·AI 모델 평가·서비스 기획에 강한 기술 제안서 작성자다. 아래 GitHub 저장소의 `main` 브랜치를 직접 읽고, 광운대학교 제2회 파이썬 SW 활용 경진대회 심화활용(Innovation) 트랙에 제출할 한국어 제안서를 작성하라.

- 저장소: https://github.com/famouxsss24/PDPH_PublicDataPythonHackathon
- 기준 브랜치: `main`
- 결과 형식: Notion에 그대로 붙여 넣을 수 있는 완성형 Markdown
- 제출 방식: Notion에서 PDF로 내보내 제출
- 공식 평가 항목:
  1. 문제 정의
  2. 데이터 분석 및 전처리
  3. AI 모델의 적합성
  4. 창의성 및 실용성
  5. 발표와 문서화

분석 과정이나 작업 계획을 답하지 말고, 검증을 마친 최종 제안서 본문만 출력하라.

## 반드시 먼저 확인할 자료

저장소를 추측으로 요약하지 말고 다음 순서로 실제 파일을 확인하라.

1. `docs/CONTEST.md` — 대회 일정·분야·평가 항목
2. `README.md` — 실행 방법과 현재 사용자 기능
3. `docs/PROJECT_STATUS.md` — 구현·데이터·검증의 단일 기준
4. `docs/COMPETITION_RISK_AUDIT.md` — 주장 가능 범위와 남은 위험
5. `docs/HEAT_CARE_PLAN.md` — 폭염 보행 기능의 의도와 평가 정렬
6. `src/gneulro/`, `pipeline/`, `api/main.py` — 실제 파이썬 구현
7. `web/js/`, `web/css/`, `web/index.html`, `web/3d.html` — 실제 서비스 흐름
8. `tests/`, `.github/workflows/ci.yml`, `vercel.json` — 검증·재현·배포 구조

문서와 코드가 충돌하면 실제 코드와 `PROJECT_STATUS.md`의 최신 변경 기록을 우선하라. 과거 기획 문서의 Leaflet, 미구현 SDI, LightGBM+SHAP 계획 등을 현재 구현으로 잘못 쓰지 마라.

## 제안서의 핵심 목적

단순 기능 소개가 아니라 아래 질문에 답하는 설득력 있는 기술 제안서를 작성하라.

- 노원구 보행에서 어떤 문제가 발생하며 기존 최단거리 지도만으로 왜 충분하지 않은가?
- 공공데이터를 어떤 계약과 파이프라인으로 정제해 실제 서비스 데이터로 만들었는가?
- 그림자·경로·현재 날씨·쉼터를 어떤 계산 원리와 책임 분리 구조로 연결했는가?
- AI 모델을 왜 그대로 제품에 넣지 않았고, 어떤 검증 게이트로 적합성을 판단했는가?
- 이 서비스가 기존의 쉼터 위치 지도나 단순 그늘 지도보다 무엇이 새롭고 실제 사용 가능한가?
- 다른 사람이 GitHub를 clone하거나 배포 URL을 열어 어떻게 결과를 재현할 수 있는가?

구현한 핵심 기술, 선택 의도, 실패한 실험, 한계, 다음 제안을 하나의 일관된 서사로 연결하라.

## 반드시 반영할 현재 구현 사실

아래 내용은 저장소에서 다시 검증한 뒤 구체적인 수치와 파일 근거를 포함해 설명하라.

### 서비스와 사용자 흐름

- 서비스 범위는 노원구 보행망이다.
- 장소·주소 검색 → 현재 날씨 확인 → 시간대별 경로 비교 → GPS 도보 안내 또는 1×·2×·4×·8×·16× 경로 미리보기 → 3D 그림자 근거 화면으로 이어진다.
- 지도 클릭 팝업은 330px 정보 카드로 출발·도착을 빠르게 지정하며, 데스크톱·모바일 안전영역을 침범하지 않도록 보정한다.
- 출발지의 Open-Meteo 모델 현재 기온·체감온도를 조회하고, 현재 체감온도 33°C 이상일 때만 외출을 미루는 문구를 표시한다.
- 33°C는 현재값에 적용한 서비스 트리거다. 기상청의 공식 폭염특보는 지속 기간과 영향도를 함께 판단하므로 이를 공식 특보라고 쓰지 않는다.
- 날씨 공급자 장애 시 외출 자제 판단을 만들지 않고 중립 문구로 폴백한다.

### 데이터와 파이프라인

- GIS건물통합정보 건물, 서울시 가로수, OSM 보행망·장소, 서울시 S-DoT, 서울시 공식 무더위쉼터를 사용한다.
- `PROJECT_STATUS.md`에 기록된 건물·가로수·격자·그래프·장소·쉼터 개수와 데이터 시점을 정확히 인용한다.
- 모든 거리·면적·그림자 연산은 미터 투영좌표계 EPSG:5186에서 처리하고, 표시·API 응답은 EPSG:4326으로 변환한다.
- `pipeline/01_prepare.py`부터 그림자, 격자, 그래프, 장소, 정적 스냅샷, 공식 쉼터로 이어지는 배치 구조를 설명한다.
- 원천 대용량 데이터는 Git에서 제외하지만, 재배포 가능한 처리 산출물은 GitHub 100MB 제한 안에서 추적해 새 clone 실행을 보장한다.
- `web/mock`은 손으로 만든 가짜 데이터가 아니라 `pipeline/10_mockdata.py`가 실제 파이프라인 산출물에서 만든 정적 스냅샷이다.

### 핵심 계산

- 건물 그림자 길이 `L = 높이 / tan(태양고도)`와 태양 방위각 반대 방향 이동 원리를 설명한다.
- 건물 footprint와 이동 footprint의 convex hull을 사용하는 1차 근사와 오목 건물 과대추정 한계를 함께 쓴다.
- 가로수는 수관 정보 부족으로 반경 3m 원형 그늘로 근사한다.
- 보행망 엣지별 시간대 그늘율을 사전 계산하고, 거리와 햇빛 노출을 결합한 가중 최단경로로 경로를 비교한다.
- 실제 보행 중 시간이 흐르는 점을 반영해 엣지 이동시간 × 햇빛 비율을 적산하는 `sun_exposure_proxy_v1`의 의도와 한계를 설명한다.
- 일반/어르신 동행 모드의 내부 sun-minute 값은 의학 기준이 아니라 상대 비교·쉼터 탐색 파라미터임을 명시한다.
- 공식 무더위쉼터는 일반 이용 가능 후보만 자동 경유 대상으로 쓰고, 운영 여부가 불명확하면 확인 필요로 표시한다.

### 아키텍처와 코드 품질

- 배치 파이프라인 → GeoParquet·GraphML 저장 → FastAPI → HTML/CSS/ES module → MapLibre 2D·3D의 단방향 구조를 설명한다.
- 핵심 로직은 `src/gneulro/`, 실행 순서는 `pipeline/`, 계약 변환은 `api/`, 화면 책임은 `web/`으로 나눈 이유를 설명한다.
- 로컬 실행은 FastAPI live API를 사용하고, Vercel은 정적 실데이터 스냅샷과 경량 `/api/weather` 함수로 공개 실행성을 확보한다.
- Ruff, pytest, Node 구문 검사, GitHub Actions 다중 OS 검증과 현재 통과 결과를 정확히 쓴다.
- Orca 내부 브라우저에서 데스크톱 1138×871, 모바일 390×844, 2D·3D 라이트/다크, 콘솔 오류, UI 경계 상자를 검증한 사실을 반영한다.

### AI 모델 적합성 — 과장 금지

이 항목을 회피하지 말고 오히려 프로젝트의 모델 의사결정 품질로 설명하라.

- 현재 제품 경로에는 LLM·비전 모델·학습 모델 호출이 없다.
- S-DoT 상대 온도 편차 모델을 센서 단위 GroupKFold로 평가했다.
- 타깃, 입력 피처, 기준선, MAE, fold별 판단, 제품 채택 게이트를 설명한다.
- 저장소에 기록된 실제 결과인 기준선 MAE, 모델 MAE, 평균 개선률, 개선 fold 수를 정확히 사용한다.
- 성능 게이트를 통과하지 못해 `eligible_for_product=false`로 두고 경로 계산에 사용하지 않은 결정을 숨기지 않는다.
- 물리 기반 그림자·그래프 계산을 AI라고 부르지 않는다.
- 이 결과를 “실패” 한 줄로 끝내지 말고, 데이터 누수 방지·공간 일반화 검증·부적합 모델 배포 차단이라는 책임 있는 AI 적용 절차로 평가한다.
- 후속 AI 제안은 같은 날짜의 기상·운량·풍속·토지피복·현장 쌍대 관측을 확보한 뒤, 기준선 대비 평균 5% 이상 개선과 과반 fold 개선을 통과할 때만 제품에 연결하는 구체적 게이트로 제시한다.

## 평가 항목별 작성 요구

### 1. 문제 정의

- 노원구의 생활 보행 상황과 햇빛 노출 문제를 구체적인 사용자 시나리오로 시작한다.
- 대상 사용자, 서비스 범위, 사용 순간, 해결하려는 의사결정을 한 문장씩 명확히 정의한다.
- “최단거리”와 “점 형태 쉼터 정보”만으로 해결되지 않는 이유를 설명한다.
- 문제 → 원인 → 필요한 데이터 → 제품 기능의 연결이 끊기지 않게 작성한다.
- 확인되지 않은 질병 예방률·기온 감소량·사용자 수요 통계는 만들지 않는다.

### 2. 데이터 분석 및 전처리

- 원천별 출처, 공간·시간 단위, 핵심 컬럼, 정제 방식, 처리 산출물을 정리한다.
- 좌표계 변환, 높이 결측 대체, 잘못된 좌표 제거, 장소 중복 제거, 보행망 스냅, 시간 보간을 기술적으로 설명한다.
- 관측값, 공공 원천 데이터, 물리 시뮬레이션, 실데이터 파생 정적 스냅샷을 서로 구분한다.
- 데이터 계약과 계층 분리가 재현성·오류 추적에 어떤 이점을 주는지 설명한다.
- 숫자는 저장소에 근거가 있는 값만 쓰고 수치 옆에 파일 또는 원천 링크를 붙인다.

### 3. AI 모델의 적합성

- 왜 S-DoT을 검토했는지, 어떤 예측 문제로 정의했는지, 공간 교차검증이 왜 필요한지 설명한다.
- 모델 결과와 채택 거부 판단을 표 하나로 간결하게 제시한다.
- 현재 제품의 핵심이 물리 모델과 그래프 최적화임을 정직하게 밝힌다.
- 향후 AI가 실제로 추가 가치를 만들 수 있는 조건과 검증 계획을 제안한다.
- “AI를 넣었다”가 아니라 “적합성을 측정하고 부적합한 모델을 차단했다”는 기술 의도를 분명히 한다.

### 4. 창의성 및 실용성

- 정적인 그늘 지도와 달리 시간에 따라 움직이는 그림자를 경로 전체에 누적하는 점을 핵심 차별점으로 제시한다.
- 경로, 출발 시각, 현재 체감온도, 공식 쉼터, 3D 근거 화면이 하나의 사용자 결정으로 연결되는 점을 설명한다.
- 검색, 모바일 UI, GPS 안내, 미리보기 배속, 다크 모드, 저사양 3D 최적화, 실패 폴백 등 실제 사용성을 구체적으로 제시한다.
- 새 clone 실행과 Vercel 공개 배포가 심사위원·시민의 접근성을 어떻게 높이는지 설명한다.
- 남은 현장 보행 검증, 실제 저사양 기기 발열·메모리, 최신 공사·횡단보도, 공식 쉼터 운영 상태 한계를 후속 계획으로 제시한다.

### 5. 발표와 문서화

- 5~10분 발표에서 보여줄 데모 순서를 5단계 이내로 제안한다.
- 질의응답에서 자주 나올 질문과 짧고 정직한 답변을 6개 이내로 제시한다.
- GitHub 실행법, 데이터 출처, API 계약, 테스트, 위험 감사 문서를 재현성 증거로 연결한다.
- 마지막에는 평가 항목별로 어떤 근거가 어느 절에 있는지 한눈에 보는 매핑 표를 넣는다.

## 구현·실험·제안 구분 규칙

문장 시제를 사용해 상태를 명확히 구분하라.

- 구현 완료: “구현했다”, “제공한다”, “검증했다”
- 실험 결과: “평가 결과”, “게이트 미통과”, “제품에서 제외했다”
- 후속 제안: “추가 확보 후 적용한다”, “다음 단계에서 검증한다”

SDI Top-10 분석 파이프라인은 구현됐지만 현재 시민용 경로 화면에는 연결되지 않았다. 이 상태를 정확히 구분하고, 실시간 실제 그림자, 기상청 공식 특보, 거리뷰 비전, LLM 브리핑, 냉각 효과 예측을 현재 기능처럼 쓰지 마라. 후속 제안으로 넣을 때는 필요한 데이터, 방법, 수용 기준을 함께 적어라.

## Notion PDF 레이아웃 규칙

공식 페이지 제한이 저장소에 없다면 A4 약 10~12쪽 밀도를 목표로 하라. 내용이 많다고 글자 크기를 줄이게 만들지 말고 부록으로 분리하라.

- 단일 열 레이아웃으로 작성한다.
- H1은 문서 제목 한 번만 사용하고, 본문은 H2·H3까지만 사용한다.
- 주요 평가 항목 전에는 `---` 구분선을 둔다.
- 한 문단은 3~5문장, 한 글머리표는 2줄 이내를 지향한다.
- 표는 최대 4열, 한 셀은 2~3문장 이내로 제한한다.
- 넓은 데이터 표는 핵심 수치만 본문에 두고 상세값은 부록으로 보낸다.
- Mermaid, HTML, 다단 레이아웃, 토글, 긴 ASCII 아키텍처를 사용하지 않는다.
- 코드 블록은 꼭 필요한 공식·의사코드만 8줄 이내로 사용한다.
- 긴 URL을 그대로 노출하지 말고 설명형 Markdown 링크로 처리한다.
- 각 표와 그림 자리 앞뒤에 해설 문장을 넣어 페이지가 갈려도 의미가 유지되게 한다.
- 저장소 그림을 사용할 경우 파일 존재 여부를 확인하고, 캡션과 출처를 한 줄로 붙인다.
- 표 제목이나 소제목만 페이지 하단에 홀로 남지 않도록 절을 짧은 단위로 나눈다.
- 부록은 독립적으로 읽혀야 하며 본문 핵심 결론을 부록에만 두지 않는다.

## 문체

- 한국어 기술 제안서 문체로 작성한다.
- 첫 등장 용어는 쉬운 말로 설명하고 이후에 약어를 사용한다.
- 과장된 광고 문구보다 문제·근거·선택 이유·검증 결과를 우선한다.
- “혁신적”, “획기적”, “완벽한” 같은 형용사는 근거 없이 사용하지 않는다.
- 파이썬 초급 팀원도 발표에서 설명할 수 있도록 공식과 알고리즘을 단계적으로 풀어 쓴다.
- 심사위원이 30초만 훑어도 핵심 수치와 차별점을 찾을 수 있게 각 절 첫 문장에 결론을 둔다.

## 최종 자체 검수

출력 전에 다음을 확인하고 문제가 있으면 스스로 수정하라.

- 다섯 평가 항목이 각각 독립된 근거와 결과를 갖는가?

- 관측·시뮬레이션·정적 스냅샷·모델 추정이 섞이지 않았는가?
- AI 모델 미채택 결과를 숨기거나 물리 계산을 AI로 포장하지 않았는가?
- 기온 저감, 질병 예방, 공식 특보, 쉼터 운영 여부를 과장하지 않았는가?
- 현재 날씨 33°C 조건과 공식 기상청 특보의 차이를 구분했는가?
- GitHub clone 실행, 테스트, Vercel 배포 구조를 재현성 근거로 제시했는가?
- 넓은 표·긴 코드·Mermaid 없이 Notion PDF에서 읽기 쉬운가?
- 제안서가 구현 보고에 그치지 않고 데이터·검증 기준이 있는 후속 제안을 포함하는가?
- 마지막 답변에 메타 설명 없이 완성된 제안서 본문만 있는가?
