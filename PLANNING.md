# 그늘로(그늘路) — 태양을 피하는 길찾기
**제2회 파이썬 SW 활용 경진대회 · 심화활용(Innovation) 트랙 — 기획 & 아키텍처 명세 v2**

> 건축물대장의 건물 높이 데이터를 '그림자 생성기'로 뒤집어,
> "최단 경로 12분 vs 그늘 82% 경로 15분"을 알려주는 폭염 보행 내비게이션.

---

## 0. 설계 철학

**"우승급 기술 스택 + 우리가 화이트보드에서 설명 가능한 코드"** — 이 두 조건을 동시에 만족한다.

- 기술은 실무 표준을 쓴다 (FastAPI, PostGIS, Docker). 포트폴리오에 "했다"고 당당히 쓸 수 있는 스택.
- 단, 모든 코드는 **팀원이 심사위원 앞에서 설명할 수 있어야 채택**한다. 설명 못 하는 코드는 넣지 않는다 (질의응답 5~10분 방어가 심사의 일부).
- 각 기술은 "전체가 아니라 어디까지만" 배우는지 명시한다. FastAPI 전부가 아니라 데코레이터+함수, SQL 전부가 아니라 SELECT+공간함수 3개.
- 모든 계층에 폴백(강등 경로)을 설계해 둔다. 우승 아키텍처가 안 되면 완성 아키텍처로 후퇴.
- claude_rules 준수: 스파게티 금지(모듈 분리+린터), 불필요한 우회 금지(오버엔지니어링 컷), GitHub엔 정리된 상태만.

## 1. 최종 산출물 — MVP 동결 (스코프 고정)

만드는 것은 **"그늘로" 웹서비스 하나**다. 아래 표는 별개 제품이 아니라 그 서비스 안의 **기능 목록**이며,
MVP(Minimum Viable Product) = "1~4 기능이 돌아가면 제출 가능한 완성품"이라는 **기준선**이다.
여기서 줄이지도 않고, MVP 완성 전에 5~6(가산 기능)에 손대지도 않는다.

> 구현 상세(함수 시그니처·파일 계약·API 스키마)는 **[SPEC.md](SPEC.md)** — 코드 구현 담당(Opus)용 명세.

| # | 산출물 | 구분 |
|---|---|---|
| 1 | 노원구 시간대별 그늘 지도 (웹) | MVP |
| 2 | 그늘길 vs 최단길 경로 비교 (지도 클릭 → 두 경로) | MVP |
| 3 | **출발 시각 추천** — "지금 vs 30분 뒤, 언제 걷는 게 덜 뜨거운가" | MVP (킬러 기능) |
| 4 | 물리 단위 테스트 + S-DoT 상관 검증(간단) | MVP (신뢰성) |
| 5 | LightGBM+SHAP 검증 고도화 | 가산점 (MVP 완성 후에만) |
| 6 | SDI 히트맵 + 그늘막 입지 Top-10 정책 리포트 | 가산점 (MVP 완성 후에만) |

> #3이 독창성의 정점: 태양이 움직이므로 **같은 길도 시각마다 그늘이 다르다**. 시간대별 사전계산 결과를 보간해 "가장 시원한 출발 시각"을 추천 — 유사 서비스 전무.

**아키텍처 서사 (발표·포트폴리오용)**: "무거운 공간 연산은 배치로 사전계산 → 공간 DB에 인덱싱 → API는 질의만 → 프론트는 표시만. 각 층이 한 가지 책임만 갖는 구조" — 관심사 분리를 실제로 구현했다는 스토리.

**역할 분배 원칙**: 로직은 100% 파이썬(대회 취지). JS는 지도 표시·클릭 이벤트 ~100줄로 제한.

## 4. 핵심 알고리즘

### ① shadowcast — 자체 그림자 엔진 (프로젝트의 독자 기술)
원리: 그림자 길이 = 높이 ÷ tan(태양고도). 중학교 삼각비를 도시 스케일로.

**계산 전제 3가지 (GIS 버그의 90%가 여기서 나온다)**
- 좌표계: translate·면적 연산은 전부 **미터 투영좌표계 EPSG:5186**에서. 위경도(4326)는 표시 전용.
- 시간: pysolar에는 **KST timezone-aware datetime**만 전달 (naive datetime은 pysolar가 거부).
- 날짜: 그림자는 계절마다 다르다 → **대표일 = 8/6(본선일)** 고정 계산, DB에 `calc_date`로 기록.

```
1. pysolar: (위도, 경도, KST aware datetime) → 태양고도 alt, 방위각 azi
2. L = 건물높이 / tan(alt)
3. 그림자 방향 벡터(5186 기준): dx = L·sin(azi+180°), dy = L·cos(azi+180°)
4. shadow = convex_hull( footprint ∪ translate(footprint, dx, dy) )
   ← ㄱ자 등 오목 건물에서 과대추정 가능. "MVP용 1차 근사"로 명시하고 보고서
      한계 절에 기록 (정밀화 = 벽면별 투영, 가산 과제)
5. 노원구 전 건물 벡터화 반복 → 시간대별(10·13·14·15·17시) 그림자 레이어
```
- 가로수: 위치 buffer(3m)를 그늘로 가산 (한계로 문서화)
- 단위 테스트로 방어: "높이 10m·고도 45° → 그림자 10m", "한낮 그림자는 북쪽", "pysolar 방위각 규약(북=0°, 시계방향) 확인" → **질의응답에서 "검증했습니다"라고 답할 수 있는 무기**

### ② 격자 그늘율
50m 격자 × 그림자 레이어 `overlay` → `그늘율 = 교차면적/격자면적`. 시간대별 컬럼으로 PostGIS 적재.

### ③ 그늘 가중 길찾기
```
비용(엣지) = 거리 × (1 + β × 햇빛노출율)      # β: 사용자 슬라이더 (0=최단, 2=그늘 우선)
같은 그래프에 weight만 바꿔 shortest_path 2회 → 최단 vs 그늘 경로 비교
```
**데이터 계약 (파이프라인 ↔ API 인터페이스 고정)**
- 엣지 그늘율은 배치에서 계산해 **GraphML 엣지 속성 `shade_10 … shade_17`** 로 저장 — API는 재계산 없이 읽기만
- 계산법: 엣지 중심선 buffer(5m) ∩ 그림자 레이어 면적비 (격자 근사보다 정확)
- 스냅핑: 사용자 클릭 좌표 → `osmnx.distance.nearest_nodes()`로 최근접 그래프 노드 매칭 (API 책임)

### ④ 출발 시각 추천 (킬러 기능)
```
1. 사용자의 경로에 대해 시간대별(사전계산 5개 레이어) 총 햇빛노출량 계산
2. 시각 사이는 선형 보간 → "노출량-시각" 곡선
3. "지금 출발: 노출 720m 땡볕 / 15:40 출발: 410m" 형태로 추천
```
- 추가 계산 비용 거의 0 (사전계산 재활용) — 적은 코드로 최대 독창성을 뽑는 지점

### ⑤ 실측 보조 검증 (심사 "AI 적합성" 대응 — 클레임 수위 조절)
기온은 그늘 외에 풍속·지표면·인공열 등 교란 요인이 많다. "증명했다"고 말하면 질의응답에서 반격당한다.
클레임은 **"그늘율과 실측 기온의 관계를 보조적으로 확인했다"** 까지만.
- MVP: S-DoT 센서 위치의 그늘율-기온 **상관분석 + 시각 통제 선형회귀** → 산점도 1장 (이것만으로 충분히 설득됨)
- 가산(MVP 완성 후): LightGBM([그늘율, 시각, 건물밀도, 가로수밀도] → 기온) + SHAP 기여도 그래프
스토리: **"물리 시뮬레이션이 계산하고, 실측 데이터가 방향을 확인해준다"** — 하이브리드 구조는 유지.

### ⑥ SDI & 그늘막 입지
`SDI = (1-그늘율) × 취약인구(고령+아동) 정규화` → 상위 격자 그리디 선택(기존 쉼터 500m 제외). p-median(PuLP)은 시간 남으면.

설계 규칙 (claude_rules② 스파게티 방지):
- **로직은 src/에만.** pipeline/은 호출만, api/는 변환만, web/은 표시만.
- 함수엔 타입힌트, 모듈 상단에 "이 파일이 하는 일" 3줄 docstring.
- 커밋 전 `ruff check` + `pytest` 통과 (rules⑥: GitHub엔 정리된 상태만).

## 6. DB 스키마 (PostGIS · 4개 테이블이면 충분)

| 테이블 | 주요 컬럼 | 용도 |
|---|---|---|
| buildings | id, geom(Polygon), height, floors, source | 원천 |
| shadows | calc_date, hour, geom(MultiPolygon), computed_at | 시간대별 그림자 레이어 (대표일 명시) |
| grid_cells | id, geom, shade_10 … shade_17, pop_vuln, sdi | 격자 마트 (조회 중심) |
| sensors / readings | id, geom / sensor_id, ts, temp | 검증용 |

- 그림자는 날짜(계절)에 따라 달라지므로 `calc_date` 필수 — "왜 8/6 기준인가"를 데이터로 답한다.
- SRID는 지오메트리 **컬럼 정의에 고정(5186)** — 행별 crs 컬럼은 두지 않는다 (PostGIS가 컬럼 수준에서 강제하므로 중복).
- 공간 질의는 3개만 쓴다: `ST_Intersects`(교차), `ST_AsGeoJSON`(변환), GiST 인덱스(자동 가속).
- 보행 그래프는 `graphml` 파일 (그래프는 파일이 정석 — DB에 억지로 넣지 않음). 엣지 그늘 속성은 4-③ 계약 참조.

## 9. 폴백 사다리 (막히면 한 칸씩 내려간다 — 프로젝트는 절대 안 죽음)
